# Check Crawl Quality

檢查爬蟲清理後的內容，從**清潔度**和 **RAG 適用性**兩個維度評估，輸出結構化品質報告。

## 設計原則

- **格式無關**：JSON/TXT/MD/CSV/SQLite/PostgreSQL 都支援
- **網站無關**：檢查邏輯通用，不依賴特定網站結構
- **抽樣優先**：大量資料時抽樣，不需全部讀完
- **區分 noise 與設計**：nav 殘留和刻意 metadata 前綴性質不同，建議不同
- **per-source 必要**：多來源資料集必須按來源分別檢查，避免問題被全域稀釋
- **可操作結論**：每個發現附帶修正方向

---

## 執行流程

### Step 1. 載入資料

`$ARGUMENTS` 可能是目錄路徑、檔案路徑/glob、DB 連線（`.db`/`.sqlite`/`sqlite:`）、或自然語言描述。

1. 判斷來源類型，偵測內容欄位（JSON 常見：`content`/`text`/`body`/`markdown`）
2. 統計總筆數；> 500 筆時抽樣 200-300 筆（分層抽樣，涵蓋不同大小區間）
3. 載入每筆：識別碼、內容全文、標題（如有）、內容長度、來源標記

### Step 2. 來源識別（必要，不可跳過）

識別來源分群（JSON `source`/`category` 欄位、子目錄名、DB source column），無法分群則視為單一來源。

**分析策略**：全域跑一次完整檢查 + 每個來源內部分別跑，合併報告。某來源 83% 有 boilerplate 但只佔總量 4%，全域看不出來 — per-source 能抓到。

**抽樣保障**：每個來源至少抽 20 筆（或全部，取較小值）。

### Step 3. 統計總覽

```
總筆數 / 抽樣數 / 來源數
各來源筆數和佔比
內容長度：min / p25 / median / p75 / max
空白或極短（< 50 字元）筆數
平均每筆連結數
```

### Step 4. 品質檢查（兩維度、八項）

每項獨立執行，某項無法執行不影響其他。每項同時在全域和 per-source 兩層級執行。

---

## A. 清潔度（有沒有 noise）

### 4a. 前綴重複（Prefix Duplication）

取每筆前 3 行，計算出現頻率。> 10% 的前綴進一步分三類：

| 類型 | 特徵 | 歸入維度 | 建議 |
|------|------|----------|------|
| **noise** | markdown 連結列表、nav 文字、與正文無關 | 清潔度扣分 | 清理階段移除 |
| **metadata** | `key: value` 或 `[key] value` 格式 | RAG 適用性（4g） | 移至 payload metadata |
| **parsing artifact** | 內容重複了 title/author 等獨立欄位 | RAG 適用性（4g） | 上傳時去除重複 |

展示 top 5 重複前綴 + 次數 + 類型 + 範例。僅 noise 扣清潔度分。

### 4b. 內容重複（Content Duplication）

取每筆前 200 字元作為指紋，統計相同指紋的文件數。> 3 個文件標記為重複群組。

### 4c. 連結密度（Link Density）

計算每筆 markdown 連結 `[...](...)` 文字佔總文字比例，標記 > 60% 的文件。

### 4d. 內容過短（Thin Content）

移除 markdown 格式後計算淨文字長度，標記 < 50 字元的文件，分兩類：

- **空佔位檔**（content 為空/僅空白）：不該納入知識庫，建議直接排除
- **薄內容**（有內容但淨文字 < 50）：改善清理邏輯或設最低門檻

Per-source 若空佔位檔 > 50%，該來源整體不適合納入。

### 4e. Web 殘留物（Web Artifacts）

偵測未清理的 web 元素，每個 pattern 統計影響筆數：

**程式碼殘留**：
- HTML 標籤：`<div`、`<span`、`<script`、`</`（`<br>` 可接受）
- JavaScript：`javascript:`、`onclick`、`window.`、`document.`、`function(`
- JS markdown 連結：`[文字](javascript:void(0))` — 爬蟲將按鈕轉成 markdown 的常見產物，獨立計數
- CSS：`{color:`、`{display:`、`{margin:`
- 編碼殘留：`&amp;`、`&lt;`、`&#`、`\u00`
- PDF 二進位：`%PDF` 開頭或含 `endstream`/`endobj` — 未經解析的 PDF 原始資料

**UI 文字殘留**（獨立成行才算）：
- `Download PDF`、`Share this`、`Back to top`、`Read more`、`Click here`、`Subscribe`

### 4f. 結構異常（Structural Anomalies）

- **標題層級跳躍**：`# h1` 後直接 `### h3`
- **孤立標題**：標題後無段落文字
- **過深巢狀**：> 3 層 list 巢狀
- **大量空行**：連續 > 5 空行
- **內容截斷**：末行 markdown 語法未閉合（如 `[文字](`）

---

## B. RAG 適用性（對檢索好不好用）

### 4g. 內容-Metadata 分離（Content-Metadata Separation）

合併檢查兩種 metadata 污染 content 的情況：

**結構化 metadata 混入**：
- 掃描前 10 行，偵測 `key: value` 格式行
- 統計有 metadata 前綴的文件比例 + metadata 行佔 content 比例
- 大量文件共享相同 metadata 前綴會使 embedding 向量在 metadata 維度聚集，降低內容區分度

**欄位冗餘（原 Parsing Artifact）**：
- 適用於有獨立 metadata 欄位的格式（JSON 有 `title`/`author`/`date` 等）
- 檢查 content 前 10 行是否重複這些欄位值
- 常見來源：PDF 標題重複、`<h1>` 同時存於 title 和 content、公文格式

### 4h. Chunk 品質（Chunk Quality）

**開頭區分度**：取每筆前 100 字元（去除 metadata 行），計算 unique 比例。< 50% 表示 embedding 區分度低。

**大小適當性**（三級）：

| 閾值 | 意義 | 建議 |
|------|------|------|
| < 100 字元 | 太短，缺乏語義 | 合併或排除 |
| > 8,000 字元 | 超過 embedding 有效上下文 | 上傳前 chunk splitting |
| > 100,000 字元 | 災難性大小，很可能是解析錯誤 | 回頭檢查爬蟲/PDF 解析邏輯 |

**自足性**（僅對有 split_index/chunk_id 的資料）：非首 chunk 以代名詞或接續語開頭（「此外」「另外」「其」「該」）表示分割點不佳。

## Step 5. 輸出報告

```markdown
## Crawl Quality Report

**資料來源**: {路徑/DB}
**總筆數**: {N} | **抽樣數**: {M}
**來源數**: {K} 個（{各來源名稱和筆數}）
**整體評估**: {一句話}

### 評分

| 維度 | 分數 | 說明 |
|------|------|------|
| 清潔度 | {0-100} | 有沒有 noise |
| RAG 適用性 | {0-100} | 對檢索好不好用 |
| **綜合** | **{0-100}** | 加權平均（清潔 60% + RAG 40%） |

### 統計總覽

（基礎統計表格）

### 問題發現

（依嚴重度排序：嚴重 → 警告 → 資訊）

#### [{嚴重度}] {問題標題}
- **維度**: 清潔度 / RAG 適用性
- **影響範圍**: {source_name} {N}筆中{M}筆 ({%}) 或 全域
- **描述**: 具體說明
- **範例**: 2-3 個具體檔案 + 內容片段
- **建議**: 修正方向

### Per-Source 摘要

（僅列有問題的來源）

| 來源 | 筆數 | 清潔度 | RAG 適用性 | 主要問題 |
|------|------|--------|-----------|----------|

### 清理建議

依優先順序列出具體修正建議。
```

## 評分標準

| 分數 | 意義 |
|------|------|
| 90-100 | 優良：可直接用於 RAG |
| 70-89 | 尚可：有改善空間，不影響大部分查詢 |
| 50-69 | 需改善：會影響檢索品質 |
| < 50 | 嚴重：需要重新處理 |

### 清潔度扣分（從 100 起扣）

全域和 per-source 取較嚴重者。Per-source 按佔總量比例加權，至少扣對應等級一半。

| 檢查 | 嚴重 | 警告 |
|------|------|------|
| 4a noise 前綴 | > 30%: -20 | 10-30%: -10 |
| 4b 內容重複 | > 20%: -20 | 5-20%: -10 |
| 4c 連結密度 | > 15%: -15 | 5-15%: -8 |
| 4d 內容過短 | > 10%: -15 | 3-10%: -8 |
| 4e Web 殘留 | > 5%: -10 | 1-5%: -5 |
| 4f 結構異常 | > 10%: -10 | 3-10%: -5 |

4a 的 metadata / parsing artifact 不扣清潔度，歸入 4g。

### RAG 適用性扣分（從 100 起扣）

| 檢查 | 嚴重 | 警告 |
|------|------|------|
| 4g metadata 混入（> 20% content 且 > 50% 文件） | -20 | 僅比例高: -10 |
| 4g 欄位冗餘 | > 80%: -10 | 50-80%: -5 |
| 4h 開頭區分度 | < 50%: -15 | 50-70%: -8 |
| 4h chunk 過短 (< 100 字) | > 10%: -10 | 3-10%: -5 |
| 4h chunk 過長 (> 8K 字) | > 5%: -10 | 1-5%: -5 |
| 4h 災難性大小 (> 100K 字) | > 1%: -5 | — |
| 4h 自足性差 | > 30%: -15 | 10-30%: -8 |

---

## 規模策略

| 筆數 | 策略 |
|------|------|
| ≤ 500 | 全部讀取 |
| 501-5000 | 抽樣 300 筆（分層抽樣，每來源至少 20 筆） |
| > 5000 | 抽樣 500 筆 + Agent 平行分析 |

## 注意事項

- **唯讀**：不修改原始資料
- **標明抽樣**：報告註明抽樣比例，結果為估計值
- **容許誤判**：提供範例讓使用者判斷（如教學文章講解 HTML 不算殘留）
- **善用 context**：使用者提供網站描述時據此調整判定
